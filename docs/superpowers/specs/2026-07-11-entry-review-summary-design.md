# 开仓总结设计

## 目标

把回测页的“自动策略”替换为“开仓总结”，用真实开仓快照回答三个问题：为什么开、开得是否合理、同类开仓下一步应该怎么优化。

该功能只生成事实和建议，不自动修改实盘规则。后续是否让建议生效，必须走独立确认流程。

## 页面结构

“开仓总结”包含两个连续区块，不再拆出更多标签页：

1. **开仓事实**：逐仓位展示开仓时间、币种、方向、分类、模板、开仓说明、结果标签、当前或最终收益、最大有利空间、最大不利回撤。
2. **分类总结**：按 `币种分类 + 入场模板` 汇总样本，展示合理、偏早、追高、条件错误、待观察数量，以及一条可执行建议。

当前持仓和历史仓位使用相同结构。当前持仓标记为“持有中”，结果随最新 K 线更新；历史仓位使用完整持仓周期和必要的平仓后走势。

现有“动作流水”独立标签页一并移除。动作流水继续作为后台证据层，在单条开仓事实或平仓事实中通过“查看动作详情”按需展开，不再让用户直接浏览大量扫描和拦截记录。

## 数据模型

新增 `trade_entry_reviews` 表，一条记录对应一个真实仓位生命周期，唯一键为 `position_trade_id`。

### 固化字段

以下字段在开仓成交后写入，此后不可被复盘任务覆盖：

- 身份：`position_trade_id`、`source_decision_id`、`symbol`、`alpha_symbol`、`side`。
- 分类：`strategy_source`、`category`、`entry_template`、`market_regime`。
- 成交：`entry_time`、`entry_price`、`quantity`、`leverage`、`margin`、`notional`。
- 风控：`stop_loss`、`stop_pct`、`take_profit_1`、`take_profit_2`、`risk_reward_ratio`、`atr_pct`。
- 评分：`total_score`、`grade`、`score_items_json`。
- 条件：`trend_score`、`breakout_state`、`spot_volume_ratio`、`futures_volume_ratio`、`volume_sync_state`、`spread_pct`、`orderbook_state`、`passed_conditions_json`、`relaxed_conditions_json`。
- 原始证据：`features_json`、`risk_params_json`、`reason_json`。
- 可读说明：`entry_reason_text`。

### 更新字段

复盘任务只更新以下结果字段：

- 状态：`position_status`、`exit_time`、`exit_price`、`net_pnl`、`pnl_pct`。
- 走势：`return_now`、`max_favorable_return`、`max_adverse_return`、`first_hit_1r_time`、`first_hit_minus_075r_time`、`bars_observed`。
- 结论：`review_label`、`review_reason`、`reviewed_at`。

`entry_snapshot_json` 保存完整不可变快照，用于兼容未来新增指标。结构化列只保存页面筛选、排序和汇总真正需要的字段。

## 数据写入

### 新开仓

交易执行确认成交后，调用 `record_entry_review_snapshot(...)`：

1. 从成交结果取得真实开仓价、数量、杠杆和保证金。
2. 从对应 `strategy_decisions` 取得分类、评分、特征、风险参数和原始原因。
3. 生成 `position_trade_id`，写入一次性快照。
4. 使用 `INSERT ... ON CONFLICT DO NOTHING`，防止重试覆盖原始事实。

快照写入失败只记录错误，不阻断已经成交的订单；后台复盘会从动作流水补写。

### 历史回填

`backfill_entry_reviews()` 从 `position_trades` 和当前持仓出发：

1. 用 `symbol + side + entry_time` 查找开仓前最近一条成功的 open decision。
2. 匹配窗口默认 15 分钟；同一仓位只绑定一条最接近的决策。
3. 有证据的字段正常填充；历史缺失字段写 `NULL`，说明文本明确显示“历史指标未记录”。
4. 不使用当前扫描值替代历史值，不虚构评分或指标。

## 开仓说明

说明由固定模板生成，不调用大模型，不堆原始 JSON。优先选取 3 至 5 个真正促成开仓的条件：

> AAVEUSDT 做多：普通币 accumulation 模板，评分 72.4；趋势确认，现货与合约同步放量，价格尚未完全突破；使用 3 倍杠杆、295U 保证金，初始止损 6.5%，因此开仓。

生成规则：

- 先说分类、模板、总分。
- 再说最关键的通过项和被放宽项。
- 最后说杠杆、保证金和初始止损。
- 缺失字段直接省略；若关键历史字段都缺失，显示“该历史仓位仅保留成交事实，开仓指标未记录”。

## 合理性判断

所有类型统一按初始风险 `R = abs(entry_price - stop_loss) / entry_price` 归一化，避免高波动 Alpha 和低波动蓝筹直接比较百分比。

- **合理**：先达到 `+1R`，且此前未达到 `-0.75R`。
- **偏早**：先达到 `-0.75R`，之后仍达到 `+1R`，说明方向可能正确但时机偏早。
- **追高/偏晚**：开仓后先达到 `-0.75R`，在观察窗口内未达到 `+0.5R`，且开仓快照显示突破过热或价格显著偏离短期均线。
- **条件错误**：历史仓位最终达到 `-1R`，并且全程未达到 `+0.5R`。
- **待观察**：当前仓位、后续 K 线不足，或走势未落入以上明确情形。

观察窗口按币种分类设置：Alpha 24 小时、普通币 48 小时、蓝筹 72 小时。历史仓位至少观察至平仓；为判断开仓方向是否最终兑现，可继续观察到分类窗口结束。

若历史仓位没有可靠初始止损，则使用开仓时 ATR 推导 R；ATR 也缺失时只展示百分比走势并标记“待观察”，不强行分类。

## 分类总结与建议

汇总键为 `strategy_source + category + entry_template`，只使用已经有明确标签的历史仓位，当前持仓只展示、不进入建议统计。

- 少于 4 笔：显示“样本不足，继续观察”。
- 4 至 7 笔：只给宽松建议，例如“可等待回踩 0 至 0.5 ATR 再入场”。
- 8 笔及以上，且同一问题占比达到 60%：给具体建议。
- 不满足稳定性条件：保持现有规则，不为了生成建议而输出建议。

建议仅来自固定规则：

- “偏早”集中：增加 1 根确认 K 线，或等待回踩不破后开仓。
- “追高/偏晚”集中：限制价格偏离，突破后等待回踩或二次放量。
- “条件错误”集中：指出失败样本共同缺失的关键条件，例如合约未同步放量或趋势分不足。
- “合理”占比高且平均 MFE 良好：保持当前入场条件。

页面同时显示样本数、问题占比和平均 MFE/MAE，使建议可核验。

## 服务协作

1. Trader 成交后立即固化开仓快照。
2. Policy loop 每轮补齐遗漏快照，并更新当前/历史仓位的走势结果。
3. API 汇总返回 `entry_reviews` 和 `entry_summaries`。
4. 前端将“自动策略”替换为“开仓总结”，移除自动候选和策略版本表格。
5. 现有自动策略内部逻辑暂时保留，避免影响其他依赖，但不再作为页面主功能；开仓总结不会自动调用其生效接口。
6. `decision_actions` 继续记录扫描、拦截、开仓、减仓、滚仓和平仓动作，为开仓总结、平仓总结及故障排查提供原始证据。

## 动作证据

动作流水不再作为面向用户的独立功能，但保留以下后台职责：

- 串联一次仓位生命周期内的入选、决策、成交、仓位调整和平仓动作。
- 为开仓说明和平仓说明提供可追溯的触发条件、执行参数及执行结果。
- 排查未开仓、重复开仓、异常减仓或错误平仓等执行问题。
- 对比策略调整前后的真实执行结果。

动作与仓位优先通过 `position_trade_id` 关联；历史数据缺少该字段时，使用 `symbol + side + 时间窗口` 回退匹配。页面上的“查看动作详情”只查询当前仓位相关动作，并按时间顺序展示动作、原因、关键参数和执行结果。

`decision_actions` 仅保留最近 5 天原始数据，每日清理一次。开仓和平仓复盘表已经固化的事实、结论和必要原因不随动作流水删除，避免清理后复盘结果失去核心信息。

## API

现有 `/api/policy-loop/summary` 增加：

- `entry_reviews`: 最近开仓事实，默认 100 条。
- `entry_summaries`: 分类与模板汇总。
- `entry_review_status`: 总样本、已评价、待观察、历史缺失快照数量。

增加按仓位读取证据的接口：

- `GET /api/policy-loop/positions/{position_trade_id}/actions`：返回该仓位最近 5 天内的相关动作。

默认汇总接口不再返回全量 `actions`，避免无意义的数据传输和数据库扫描。

“立即复盘”同时执行开仓和现有平仓复盘。复盘失败时保留上次结果，并返回独立错误字段，不让整个回测页加载失败。

## 性能与一致性

- `trade_entry_reviews(position_trade_id)` 唯一索引。
- `trade_entry_reviews(entry_time DESC)` 页面排序索引。
- `trade_entry_reviews(strategy_source, category, entry_template, review_label)` 汇总索引。
- 每轮只处理新增仓位和未完成评价的仓位，不全表扫描。
- 快照字段不可更新；结果字段使用单独的更新语句。
- 开仓事实来源优先级：真实成交 > 开仓决策 > 历史重建；来源写入 `snapshot_source`。
- `decision_actions` 每日清理一次，只保留最近 5 天；复盘摘要不受该清理影响。

## 测试与验收

### 后端

- 新开仓能完整保存评分、条件、杠杆、保证金和止损快照。
- 重复同步不会覆盖首次快照。
- 历史匹配只选择开仓前 15 分钟内最近决策。
- 缺失历史指标时不使用当前数据补齐。
- 五类标签的边界测试覆盖 `R` 计算和先后顺序。
- 当前持仓不参与分类建议。
- 样本阈值 4、8 和问题占比 60% 的建议测试。
- API 在开仓复盘失败时仍返回平仓数据和上次开仓结果。

### 前端

- “自动策略”不再出现，“开仓总结”可正常切换。
- “动作流水”独立标签页不再出现，开仓和平仓事实可按仓位展开动作详情。
- 当前和历史仓位状态清晰区分。
- 一句话说明、标签、建议和缺失字段在窄屏下不溢出。
- 空数据时显示“暂无开仓总结”，不显示误导性的 0 值建议。

### 完成标准

1. 任意新开仓在一次交易循环后可看到真实开仓说明。
2. 现有历史仓位能回填成交事实，有证据才展示指标。
3. 当前和历史仓位都能得到可解释结果或明确“待观察”。
4. 分类建议能追溯到样本数量和问题占比。
5. 开仓总结只提供建议，不会未经确认改变实盘开仓规则。
